**Part A: Create `.claude/skills/fixme/agents/browser-verifier.md`**

New agent file following the investigation-agent.md pattern. Structure:

Frontmatter:
```yaml
---
name: browser-verifier
description: "Re-runs reproduction steps to verify bug is fixed after code changes"
tools: Read, Write, Edit, Bash(playwright-cli:*), Bash(mkdir *), Bash(node ~/.claude/skills/fixme/scripts/fixme-tools.cjs *), Glob, Grep
model: inherit
skills:
  - playwright-cli
---
```

Identity: "You are the browser verifier. You confirm that a fix actually resolves the bug by re-running the original reproduction steps in a real browser."

Input: 3 parameters via Task prompt -- ticket folder path, project context path, dev server URL.

6-phase workflow:

**Phase 1: Read Ticket** -- Read ticket.md from ticket folder. Extract from `<!-- section: investigation -->`: reproduction steps (numbered list under `#### Reproduction Steps`), reproduction evidence, affected URL. Extract from `<!-- section: structured-fields -->`: expected behavior, actual behavior.

**Phase 2: Navigate and Reload** -- `playwright-cli open <affected-url>`. Wait for page ready: `playwright-cli run-code "async page => { await page.waitForLoadState('networkidle'); }"`. Wait 2-3 seconds for HMR to settle. Take baseline snapshot.

**Phase 3: Execute Reproduction Steps** -- For each step from the investigation section, execute via playwright-cli commands (click, fill, type, press, etc.). After each interaction, take a snapshot. Capture console errors and network failures.

**Phase 4: Determine Verdict** -- PASS: the reported bug symptom is gone AND the expected behavior is present (not just "no error"). FAIL: bug symptom still present OR expected behavior not achieved. Write a clear one-paragraph explanation of what was observed.

**Phase 5: Write Evidence** -- Create asset directory if needed: `mkdir -p <ticket-folder>/assets/`. Take screenshot: `playwright-cli screenshot --filename=<ticket-folder>/assets/verify-<descriptive-name>.png`. Write verification report to `<ticket-folder>/verifications/<NNNN>-browser-verify-<attempt>.md` with: what was tested, steps executed, observations, verdict, screenshot path. Use Edit tool to append to ticket's `<!-- section: verification -->` section with a structured `### Browser Verification (Attempt N)` subsection containing: verdict, observations, screenshot reference.

**Phase 6: Return Summary** -- Single one-liner:
- PASS: `"Browser verified #NNNN: PASS -- <what correct behavior was observed>"`
- FAIL: `"Browser verified #NNNN: FAIL -- <what's still wrong>"`

Rules (following investigation-agent.md pattern):
1. NEVER modify source code. Read-only for source. Write only to ticket verification section and assets/verifications directories.
2. Use `playwright-cli` for ALL browser interaction. NEVER use MCP tools.
3. Write evidence progressively to ticket.
4. Screenshots MUST use `--filename=<ticket-folder>/assets/verify-<descriptive-name>.png`.
5. Respect the investigation section's reproduction steps as the source of truth for what to re-test.
6. PASS requires both: bug symptom gone AND expected behavior present. "No error" alone is insufficient.

**Part B: Update SKILL.md dispatch loop**

Replace the current step 5d fixer result handling (lines around 229-237 that say "Awaiting browser verification (Phase 5)") with the full browser verification + commit + re-entry flow:

After fixer returns success ("Fixed #NNNN"):

**i. Transition to verifying:**
```bash
node ~/.claude/skills/fixme/scripts/fixme-tools.cjs ticket transition <ticket-path> verifying
```

**ii. Dispatch browser-verifier via Task tool:**
```
First, read ~/.claude/skills/fixme/agents/browser-verifier.md for your role instructions.

Verify this fix in the browser:
- Ticket folder: <ticket-folder-dir>
- Project context: .fixme/project-context.yaml
- Dev server URL: <dev_server.url from project context>
```

**iii. After browser verifier returns, read ticket state from disk.**

**iv. Handle browser verification result:**

- If "Browser verified #NNNN: PASS":
  a. Read ticket frontmatter to get `files_changed` and `slug`. Derive title from slug: replace hyphens with spaces, lowercase. Commit message format: `fix: <title>` (no ticket number, no body -- per user decision).
  b. Stage only the fix files: `git add <file1> <file2> ...` from `files_changed`.
  c. Create commit: `git commit -m "fix: <title>"`.
  d. Capture commit hash: `git rev-parse HEAD`. Use Edit to set `commit_hash:` in ticket frontmatter.
  e. Transition: `ticket transition <ticket-path> done`.
  f. Report to user: "Fixed and committed #NNNN: <summary>"

- If "Browser verified #NNNN: FAIL":
  a. Read ticket frontmatter for `current_attempt` and `max_attempts`.
  b. If `current_attempt < max_attempts`:
     - Transition: `ticket transition <ticket-path> investigating --reason "<browser failure summary>"`
     - Then transition: `ticket transition <ticket-path> fixing`
     - Re-dispatch fix-agent via Task tool (same as step 5b) but with added context: `- Browser verification failure: <ticket-folder>/verifications/<NNNN>-browser-verify-<N>.md`
     - After fix-agent returns: go back to step 5d to re-evaluate the result (the fix->verify->commit/retry cycle)
  c. If `current_attempt >= max_attempts`:
     - Read `base_commit` from ticket frontmatter.
     - Revert: `git checkout <base_commit> -- .` then `git clean -fd --exclude=.fixme/`
     - Transition: `ticket transition <ticket-path> failed --reason "Browser verification failed after all attempts: <last failure>"`
     - Report to user: "Failed to fix #NNNN after all attempts. Changes reverted."

Also add `browser-verifier.md` to the References section at the bottom of SKILL.md.
