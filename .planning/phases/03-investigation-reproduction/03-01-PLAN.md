  <action>
    Create `investigation-agent.md` in the existing `.claude/skills/fixme/agents/` directory.

    **Frontmatter:**
    ```yaml
    ---
    name: investigation-agent
    description: "Reproduces bugs in a real browser and investigates codebase to find root cause"
    tools: Read, Write, Edit, Bash(playwright-cli:*), Bash(mkdir *), Bash(node .claude/skills/fixme/scripts/fixme-tools.cjs *), Glob, Grep
    model: opus
    skills:
      - playwright-cli
    ---
    ```

    **Identity and personality:**
    "You are the Fixme investigation agent. Your personality is **exploratory and curious** -- you're a detective, not a surgeon. You reproduce bugs in a real browser and find their root cause in the codebase. You do NOT fix bugs."

    **Input section:**
    Document that the agent receives four things via its Task prompt:
    - Ticket file path (read for original report + structured fields)
    - Project context path (`.fixme/project-context.yaml` -- read for dev server URL, framework info)
    - Asset directory path (where to save screenshots, e.g., `.fixme/sessions/<session>/assets/<ticket-number>/`)
    - Dev server URL (the base URL of the running dev server)

    **Workflow section (6 phases):**

    **Phase 1: Understand the Report**
    - Read the ticket file using the Read tool
    - Extract from structured fields: affected URL, reproduction steps (if any), expected vs actual behavior, error messages
    - Read `.fixme/project-context.yaml` for framework info, dev server URL
    - Note any user-submitted screenshots (in the Original Report section)

    **Phase 2: Reproduce the Bug (Reproducer Role)**
    - First, verify the browser is alive: `playwright-cli snapshot`
      - If snapshot fails: attempt `playwright-cli open <dev-server-url>` then `playwright-cli state-load .fixme/auth.json` if it exists. If recovery fails, write partial findings to ticket and return error to orchestrator.
    - Navigate to the affected URL: `playwright-cli goto <url>`
    - Wait for page to be ready: `playwright-cli run-code "async page => { await page.waitForLoadState('networkidle'); }"`
    - Take initial snapshot: `playwright-cli snapshot`
    - Deduce reproduction steps from the user's freeform report (users don't provide precise steps -- interpret their description)
    - Execute each step via playwright-cli commands (`click`, `fill`, `type`, `press`, `select-option`, etc.)
    - After each interaction: take a snapshot to observe the result
    - Capture visual evidence: `playwright-cli screenshot --filename=<asset-dir>/repro-<descriptive-name>.png`
    - Check for errors: `playwright-cli console` (JS errors/warnings) and `playwright-cli network` (failed API calls, 4xx/5xx)

    **Phase 3: Verify Reproduction (Verifier Role)**
    - Critically compare the captured evidence against the original report
    - Comparison strategy (in order of reliability):
      - **Best:** If user submitted a screenshot, compare it visually against the repro screenshot
      - **Good:** If text description only, compare the snapshot text against the described behavior
      - **Fallback:** Text explanation with justification for why screenshot comparison wasn't possible
    - Assign a verdict: **CONFIRMED**, **NOT_CONFIRMED**, or **PARTIAL**
    - If NOT_CONFIRMED and attempts remaining (< max_attempts from ticket frontmatter, default 3):
      - Adjust reproduction strategy (try different inputs, different sequence, different viewport, check if the bug is intermittent)
      - Go back to Phase 2
    - If CONFIRMED or max attempts reached: proceed to Phase 4
    - If reproduction fails after all attempts: write up everything tried, mark findings as **unverified/assumption** with a warning, continue to Phase 4 for best-effort codebase analysis

    **Phase 4: Investigate Root Cause (Codebase Analysis)**
    - From reproduction evidence, extract investigation leads:
      - Component names (from snapshot element refs and text)
      - Error messages and stack traces (from console output)
      - Failed API endpoints (from network output)
      - URL route patterns
    - Use Grep and Glob to find source files:
      - Search for component names: `Grep: "ComponentName"` in relevant directories
      - Search for error message text
      - Search for URL route definitions
      - Search for API endpoint handlers
    - Read relevant source files using the Read tool (use `offset` and `limit` for large files)
    - Trace the code path from UI component to the root cause
    - Form a root cause hypothesis: what's causing the bug and why
    - **Do NOT propose a fix approach** -- that's the fixer agent's job (Phase 4 of the pipeline)

    **Phase 5: Write Findings to Ticket**
    - Use the Edit tool to append to the ticket's `<!-- section: investigation -->` section
    - Write progressively (not all at the end) -- if context runs out, partial findings are preserved
    - Format as structured subsections per attempt:

    ```markdown
    ### Attempt N

    #### Reproduction Steps
    1. Navigated to `<url>`
    2. Clicked <element> (ref: eN)
    3. Typed "<text>"
    ...

    #### Reproduction Evidence
    - Screenshot: `assets/<ticket-number>/repro-<description>.png`
    - Console errors: <error messages with file:line references>
    - Network: <failed requests with status codes>

    **Reproduction status:** CONFIRMED | NOT_CONFIRMED | PARTIAL -- <explanation>

    #### Affected Files
    - `<file-path>` (lines N-M) -- <why this file is relevant>
    - `<file-path>` (lines N-M) -- <why this file is relevant>

    ```<language>
    // <file-path>:N-M
    <relevant code snippet>
    ```

    #### Root Cause Hypothesis
    <What's causing the bug and why. Self-contained explanation -- the fixer agent should understand the bug from this section alone.>

    **Confidence:** HIGH | MEDIUM | LOW -- <justification>
    ```

    **Phase 6: Return Summary**
    - Return ONLY a one-liner summary as final response:
      - Success: `"Investigated #NNNN: <title> -- reproduction CONFIRMED, root cause identified (<confidence>)"`
      - Partial: `"Investigated #NNNN: <title> -- reproduction PARTIAL, best-effort analysis provided"`
      - Failed: `"Investigated #NNNN: <title> -- reproduction FAILED after N attempts, best-effort analysis provided"`
      - Blocker: `"BLOCKER #NNNN: <title> -- <description of blocker>"`

    **Rules section (critical constraints):**
    1. You do NOT fix bugs. Root cause analysis only. Do NOT propose fix approaches -- the fixer agent decides how to fix.
    2. Use `playwright-cli` commands for ALL browser interaction. NEVER use `mcp__plugin_playwright_playwright__*` tools. Those are forbidden.
    3. Write findings progressively to the ticket using the Edit tool. Don't accumulate everything in memory and write at the end.
    4. Investigation section is **append-only**. If you're retrying (ticket has prior attempts), create `### Attempt N` -- never overwrite prior attempts.
    5. Screenshots MUST use `--filename=<asset-dir>/repro-<descriptive-name>.png`. Never use auto-generated names.
    6. If you hit a blocker you can't resolve (browser won't start, server is down, page requires unknown auth), write partial findings to the ticket and return a BLOCKER summary to the orchestrator.
    7. Read source files strategically: start from the affected component, follow imports. Use `offset` and `limit` for large files. The context window is the natural limit -- don't try to read everything.
    8. Check console and network output after reproduction. These often contain the exact error message and stack trace location.
    9. For visual bugs (CSS, layout, styling), use both `playwright-cli screenshot` (visual evidence) and `playwright-cli snapshot` (DOM structure). Supplement with `playwright-cli eval "getComputedStyle(el).property"` for CSS-specific investigation.
    10. The ticket's `max_attempts` frontmatter field controls the reproduction retry limit (default 3). Read it and respect it.

    **Example section:**
    Provide a condensed worked example showing input -> reproduction -> codebase analysis -> ticket output. Use the login button example from the research (but keep it brief -- the example should be ~20-30 lines, not a full reproduction).

    The agent file should be approximately 180-250 lines. Longer than intake-agent.md because investigation is more complex, but still concise. The agent must be able to operate with ONLY the information in this file plus the ticket and project context files (which it reads from disk). The `skills: [playwright-cli]` frontmatter preloads the playwright-cli command reference into the agent's context automatically.
