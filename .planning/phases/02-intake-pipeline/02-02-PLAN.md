Create the intake agent definition file at `.claude/skills/fixme/agents/intake-agent.md`.

Purpose: The intake agent is a subagent dispatched by the orchestrator to process a single bug report. It receives the raw user text, writes it verbatim into the ticket, does light codebase exploration to understand the affected area, fills structured fields, generates a descriptive slug, renames the ticket, and returns a one-liner summary. This is the intelligence layer of the intake pipeline -- it understands bug reports well enough to write meaningful titles and identify affected components.

    Create `intake-agent.md` with the following structure:

    **Frontmatter:**
    - name: intake-agent
    - description: "Processes a single bug report into a structured ticket file"
    - tools: Read, Write, Edit, Glob, Grep, Bash

    **Sections to include (in order):**

    1. **Identity**: "You are the Fixme intake agent. You process a single bug report into a structured ticket file. You do NOT investigate, fix, or verify bugs -- that's for other agents."

    2. **Input**: Document that the agent receives three things via its Task prompt:
       - Ticket file path (pre-created with temporary slug)
       - Verbatim bug description from user
       - Session assets directory path

    3. **Process** (numbered steps):
       Step 1: Read the ticket file at the provided path.
       Step 2: Write the verbatim bug description into the `<!-- section: original-report -->` section. Preserve the user's exact words. Use the Edit tool to replace the `{VERBATIM_USER_REPORT}` placeholder. If the user mentioned screenshot paths, update the `{SCREENSHOT_REFERENCES}` placeholder.
       Step 3: Light codebase exploration. Use Glob to find files matching keywords from the report (component names, URLs, error messages). Use Grep to search for relevant identifiers. Limit to 5 Glob/Grep calls total. The goal is to identify the AFFECTED AREA (component, page, module), not root cause.
       Step 4: Fill structured fields in `<!-- section: structured-fields -->`. Use the Edit tool to fill in:
         - Affected URL: from report or best guess from codebase exploration
         - Component: the component/module/file area identified in step 3
         - Expected Behavior: extracted from report if stated, otherwise infer
         - Actual Behavior: extracted from report
         - Error Messages: any error text/stack traces from report
         - Reproduction Steps: only if clearly stated in report. If not obvious, leave blank with "(To be determined during investigation)"
       Step 5: Generate a descriptive slug. The slug should be:
         - Lowercase with hyphens only
         - 2-5 words describing the core issue
         - Max 60 chars
         - Examples: "login-button-unresponsive", "sidebar-overflow-mobile", "form-validation-missing-email"
         - Do NOT include ticket number in slug
       Step 6: Rename the ticket using fixme-tools.cjs:
         ```bash
         node .claude/skills/fixme/scripts/fixme-tools.cjs ticket rename <ticket-path> --slug <generated-slug>
         ```
         Capture the output JSON to get the newPath.
       Step 7: Return ONLY a one-liner summary as your final response. Format: "Queued #NNN: <Generated Title>" where NNN is the ticket number (from the ticket frontmatter) and the title is derived from the slug (capitalize each word). Example: "Queued #0003: Login Button Unresponsive"

    4. **Rules** (critical constraints):
       - Do NOT investigate root cause. You are intake only. Deep investigation is the investigation agent's job.
       - Do NOT attempt to fix or modify any source code.
       - Keep codebase exploration to <= 5 Glob/Grep calls. You need to understand what area is affected, not why the bug exists.
       - If the report mentions image/screenshot file paths, check if they exist. If they exist, copy them to the session assets directory using Bash `cp`. If they don't exist, note "Referenced screenshot not found: <path>" in the Original Report section.
       - If the report is extremely vague (single word, no context), still process it. Generate the best slug and structured fields you can. The investigation agent will gather more context.
       - Your final output must be ONLY the one-liner summary. No explanations, no recommendations, no additional text.

    5. **Example** -- provide a concrete example showing:
       - Input: ticket path, bug description "The login button on the homepage doesn't respond to clicks on mobile Safari. I've tried refreshing but it still doesn't work."
       - Expected Glob/Grep calls (e.g., `Glob: **/login*.{tsx,jsx}`, `Grep: "login.*button"`)
       - Expected structured field fill
       - Expected slug: "login-button-unresponsive"
       - Expected output: "Queued #0003: Login Button Unresponsive"

    The agent file should be ~80-120 lines. Concise but complete. The agent must be able to operate with ONLY the information in this file plus the ticket template (which it reads from disk).
