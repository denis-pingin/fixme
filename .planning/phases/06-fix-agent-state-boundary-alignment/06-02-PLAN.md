**New structure:**

Step 1: Read Ticket
- Read `<ticket-folder>/ticket.md`
- Extract: `number`, `slug`, `max_attempts`, `max_timeout_minutes`
- Do NOT extract `max_verify_cycles` (removed)

Step 2: Record Base State and Start Time
- Record git HEAD as base commit: `git rev-parse HEAD`
- Record wall clock time as fix_start_time
- Use Edit to set `base_commit:` in ticket frontmatter

Step 3: Dispatch Fix-Researcher
- Same dispatch prompt as current Step 4
- Same HARD GATE (verify research file exists via Glob)
- Same recording (duration, fix section bullet)
- Note: The researcher owns the `investigating -> researching` transition

Step 4: Attempt Loop (1..max_attempts)
- Before each attempt: check timeout against max_timeout_minutes
- If timeout: return failure immediately (no revert, no state transition)

  4a. Dispatch Fix-Planner
  - On first attempt: standard dispatch prompt
  - On retry (attempt 2+): include path to last verification report AND transition reason text
    The dispatch prompt for retry MUST include:
    ```
    - Transition reason: "<1-2 sentence summary from verifier's FAIL findings>"
    ```
    The planner needs this for the `verifying -> planning --reason "..."` command.
  - HARD GATE: verify plan file exists
  - Record duration, append to fix section
  - Note: The planner owns `researching -> planning` (first) or `verifying -> planning` (retry)

  4b. Dispatch Fix-Implementer
  - Same dispatch as current
  - Record duration, append to fix section
  - Capture files_changed via `git diff --name-only <base_commit> HEAD`
  - Note: The implementer owns `planning -> implementing`

  4c. Dispatch Fix-Verifier
  - Same dispatch as current (but no cycle number -- just attempt number)
  - HARD GATE: verify report file exists
  - Read verification report
  - Note: The verifier owns `implementing -> verifying`

  4d. Check Verdict
  - **PASS:** Append PASS bullet to fix section. Go to Step 5 (return success).
  - **FAIL:** Append FAIL bullet to fix section.
    - Extract failure summary from verification report (1-2 sentences of what failed and why)
    - If attempts remaining: continue loop (next iteration dispatches planner with retry feedback)
    - If attempts exhausted: go to Step 5 (return failure)

Step 5: Return Structured Result
- On success: `{ status: "fixed", ticket_path: "...", commit_hash: null, attempts: N, duration: seconds, summary: "..." }`
- On failure: `{ status: "failed", ticket_path: "...", commit_hash: null, attempts: N, duration: seconds, summary: "..." }`
- On timeout: same as failure with summary mentioning timeout
- Note: fix-agent does NOT revert, does NOT transition to failed, does NOT commit. SKILL.md handles all of that.
