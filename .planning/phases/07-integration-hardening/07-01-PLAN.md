<feature>
  <name>max_attempts enforcement in ticketTransition</name>
  <files>.claude/skills/fixme/scripts/fixme-tools.cjs, .claude/skills/fixme/scripts/fixme-tools.test.cjs</files>
  <behavior>
    When `ticketTransition()` processes a `verifying -> planning` transition:
    1. Read `current_attempt` and `max_attempts` from ticket frontmatter
    2. If `current_attempt >= max_attempts`, reject with error: `"Retry limit reached: attempt N of N (max_attempts). Transition verifying -> planning denied."`
    3. If `current_attempt < max_attempts`, allow the transition (existing behavior unchanged)

    Note: `current_attempt` starts at 0 and is incremented DURING the `verifying -> planning` transition (line ~733-735 of fixme-tools.cjs). The check must happen BEFORE the increment. So the guard is: if `(current_attempt + 1) > max_attempts` reject. Alternatively, since `current_attempt` is 0-indexed and `max_attempts` defaults to 3, the check `current_attempt + 1 >= max_attempts` means "this would be the Nth attempt and we only allow max_attempts". Simplest correct check: `(fm.current_attempt || 0) + 1 > (fm.max_attempts || 3)` — reject if the NEXT attempt number would exceed the limit. But since `current_attempt` starts at 0 (first attempt) and gets incremented to 1 on first retry, the semantics are: `current_attempt` = number of retries completed. So `current_attempt >= max_attempts - 1` means no more retries allowed. Use: `(fm.current_attempt || 0) >= (fm.max_attempts || 3) - 1`.

    Actually, let's trace the exact semantics:
    - First attempt: current_attempt = 0 (set at ticket create)
    - After first verify fail, verifying->planning increments to 1 (this is retry 1)
    - After second verify fail, verifying->planning increments to 2 (this is retry 2)
    - With max_attempts = 3: first attempt + 2 retries = 3 total attempts
    - So reject when current_attempt >= max_attempts - 1 (i.e., >= 2 for max_attempts=3)
    - The increment happens AFTER validation, so check pre-increment value: `current_attempt >= max_attempts - 1`

    Test cases:
    - current_attempt=0, max_attempts=3 -> ALLOW (this retry makes attempt 2 of 3)
    - current_attempt=1, max_attempts=3 -> ALLOW (this retry makes attempt 3 of 3)
    - current_attempt=2, max_attempts=3 -> REJECT (would be attempt 4 of 3)
    - current_attempt=0, max_attempts=1 -> REJECT (would be attempt 2 of 1)
    - current_attempt=0, max_attempts=2 -> ALLOW (this retry makes attempt 2 of 2)
    - current_attempt=1, max_attempts=2 -> REJECT (would be attempt 3 of 2)
  </behavior>
  <implementation>
    In `ticketTransition()`, after the valid-transitions check and before the reason check, add:

    ```javascript
    // Enforce max_attempts on retry transition
    if (currentState === 'verifying' && newState === 'planning') {
      const currentAttempt = fm.current_attempt || 0;
      const maxAttempts = fm.max_attempts || 3;
      if (currentAttempt >= maxAttempts - 1) {
        return error(
          `Retry limit reached: attempt ${currentAttempt + 1} of ${maxAttempts} (max_attempts). ` +
          `Transition verifying -> planning denied.`
        );
      }
    }
    ```

    This goes at approximately line 685 (after the `validNext.includes` check, before the reason check).

    Tests follow existing patterns in fixme-tools.test.cjs: use `createTicketFolder()` to make tickets, walk them to `verifying` state, then test the transition.
  </implementation>
