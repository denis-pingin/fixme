Create `~/.claude/skills/fixme/SKILL.md` with frontmatter:
```yaml
---
name: fixme
description: "Bug fix session orchestrator. Start a bug-fixing session to report, track, and fix bugs in your web application."
disable-model-invocation: true
allowed-tools: Read, Write, Bash, Task, Glob, Grep
argument-hint: "[start|resume|status|stop] [session-name]"
---
```

The SKILL.md body is the orchestrator's complete instruction set. It must contain:

**1. Role Definition:**
- You are the Fixme orchestrator. You manage bug-fixing sessions.
- You dispatch subagents for investigation, fixing, and verification.
- You NEVER do investigation, fixing, or verification yourself.

**2. Argument Parsing:**
- Parse `$ARGUMENTS` for sub-command: start (default), resume, status, stop
- Optional second argument: session name (for start/resume)

**3. Session Start Flow (new session):**
- Create session: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs session create .fixme/sessions [--name <name>]`
- Check for project context: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs context load`
  - If not found: detect -> present to user for confirmation -> save
  - If found: silently use it
- If user provided a bug report with the /fixme command: create first ticket and begin dispatch
- If no bug report: inform user the session is ready, ask them to describe a bug

**4. Session Resume Flow:**
- List sessions: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs session list .fixme/sessions`
- Find session by name (or most recent if no name given)
- Check for queued tickets: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs ticket next <session-dir>`
- If queued tickets exist: continue dispatch loop
- If no queued tickets: inform user all caught up, ask for new bug

**5. Dispatch Loop:**
- Find next ticket: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs ticket next <session-dir>`
- Transition ticket: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs ticket transition <path> investigating`
- Dispatch to implementation subagent via Task tool:
  - Pass the ticket file path (subagent reads it with fresh context)
  - Pass the project context file path
  - Subagent handles: investigate -> fix -> verify cycle
- On subagent return: read ticket status from disk (never trust memory)
- Report result to user
- Loop: check for next queued ticket

**6. Bug Intake (in-session):**
- When user describes a bug during an active session:
  - Create ticket: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs ticket create <session-dir> --slug <slug>`
  - Write the user's verbatim report into the ticket's Original Report section
  - Continue dispatch loop (new ticket is queued, will be picked up in order)
- Detect bug-like messages: look for problem descriptions, error reports, "X is broken/not working" patterns
- One bug at a time (v1) -- if user describes multiple bugs, ask them to submit one at a time

**7. Session Control:**
- "stop" / "end session": graceful stop
  - Finish current ticket (let subagent complete)
  - Run session summary: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs session summary <session-dir>`
  - Display summary to user
- "stop now" / "abort": immediate stop
  - Transition current ticket to failed: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs ticket transition <path> failed --reason "Session aborted by user"`
  - Run session summary
  - Display summary

**8. Status Query:**
- When user asks for status or types "status":
  - List tickets: `node ~/.claude/skills/fixme/scripts/fixme-tools.cjs ticket list <session-dir>`
  - Format as table: number, slug, state
  - Show current session stats

**9. Critical Rules (in a ## CRITICAL RULES section):**
- NEVER investigate bugs yourself. Dispatch to subagents.
- NEVER read ticket bodies. Only read frontmatter status via fixme-tools.cjs.
- ALWAYS read state from disk after subagent returns. Never trust in-memory state.
- Pass ticket FILE PATHS to subagents -- they read with fresh context.
- Keep your context lean. You are a dispatcher, not an implementer.
- On any fixme-tools.cjs error, report it to the user and ask how to proceed.

**10. Reference Links:**
- Point to `~/.claude/skills/fixme/references/state-machine.md` for state transition rules
- Point to `~/.claude/skills/fixme/references/project-context-schema.md` for context format

The SKILL.md should be comprehensive enough that the orchestrator can run a complete session without needing additional instructions. But it should NOT contain the actual state machine rules (those live in references/) or template content (those live in templates/).

**3. `~/.claude/skills/fixme/references/state-machine.md`**

Comprehensive reference document for the state machine. This is consumed by agents and by the orchestrator.

Contents:
- **States:** List all 7 states with descriptions (queued, investigating, fixing, verifying, done, failed, skipped)
- **Transition Matrix:** Visual table showing from-state -> valid to-states
- **Terminal States:** done, failed, skipped -- no transitions out
- **Reason Requirements:** Which transitions require a `--reason` flag (failed, skipped, verifying->investigating retry)
- **Retry Semantics:** verifying->investigating increments `current_attempt`, is bounded by `max_attempts`
- **Transition Log Format:** `{ from, to, timestamp, reason }` appended to `transitions` array
- **Duration Tracking:** `durations` object with `{ entered, exited, seconds }` per state
- **Enforcement:** Hard errors on invalid transitions. The tool (`fixme-tools.cjs ticket transition`) is the ONLY way to change state. Agents must never edit frontmatter directly.
- **Timeout Model:** Timeouts are on agent steps (configurable per-agent), not on ticket states. Step timeout -> ticket fails -> next ticket.

Include the transition matrix as both a table and a mermaid-style text diagram:
```
queued -----> investigating -----> fixing -----> verifying -----> done
  |              |                    |              |
  +-> skipped    +-> skipped          +-> failed     +-> investigating (retry)
                 +-> failed                          +-> failed
```

**4. `~/.claude/skills/fixme/references/project-context-schema.md`**

Reference document for the project context YAML schema.

Contents:
- **Purpose:** Operational project context for agents (not coding conventions -- those come from CLAUDE.md)
- **Storage:** `.fixme/project-context.yaml` (per-project, shared across sessions)
- **Schema:** Document every field in the YAML structure with types and examples:
  - `dev_server.command` (string): e.g., "yarn dev"
  - `dev_server.url` (string): e.g., "http://localhost:3000"
  - `dev_server.hmr` (boolean): true if HMR supported
  - `build.command` (string)
  - `test.runner` (string): vitest, jest, mocha, or null
  - `test.command` (string)
  - `test.filter_by_file` (string): pattern with `{file}` placeholder
  - `test.filter_by_name` (string): pattern with `{name}` placeholder
  - `test.init_command` (string|null): e.g., "docker compose up -d"
  - `lint.command` (string)
  - `framework` (string): detected framework name
  - `detected_from` (string[]): source files used for detection
  - `detected_at` (string): ISO timestamp
- **Detection Sources:** Table of where each field is detected from (package.json scripts, config files, .env)
- **Lifecycle:**
  - First run: auto-detect -> user confirms -> save
  - Subsequent runs: load silently
  - Agent corrections: propose update -> user confirms -> save
  - No silent writes ever
- **Example:** Complete example YAML for a Next.js project
