  <name>Task 1: Build fixme-tools.cjs with ticket and session subcommands</name>
  <files>~/.claude/skills/fixme/scripts/fixme-tools.cjs</files>
  <action>
Create `~/.claude/skills/fixme/scripts/fixme-tools.cjs` as a CommonJS Node.js CLI tool (shebang: `#!/usr/bin/env node`). Follow the same architectural pattern as GSD's `gsd-tools.cjs`: subcommand routing via `process.argv`, JSON output by default, `--raw` flag for plain text.

**Subcommands to implement:**

1. `ticket create <session-dir> --slug <slug> [--max-attempts N]`
   - Reads the ticket template from `~/.claude/skills/fixme/templates/ticket.md`
   - Scans `<session-dir>/tickets/` to determine the next sequential 4-digit number (0001, 0002, ...)
   - Replaces template placeholders: `{NUMBER}`, `{SLUG}`, `{SESSION}`, `{TIMESTAMP}`, `{TITLE}` (derive title from slug: hyphens to spaces, title case)
   - Writes file to `<session-dir>/tickets/NNNN-slug.md`
   - Sets initial state to `queued`, records `created` and `updated` timestamps
   - Initializes `durations: { queued: { entered: <now> } }`
   - Returns JSON: `{ path, number, slug, state }`

2. `ticket transition <ticket-path> <new-state> [--reason "..."]`
   - Reads ticket file, extracts YAML frontmatter
   - Validates transition against the strict transition matrix:
     ```
     queued -> [investigating, skipped]
     investigating -> [fixing, skipped, failed]
     fixing -> [verifying, failed]
     verifying -> [done, investigating, failed]
     done -> [] (terminal)
     failed -> [] (terminal)
     skipped -> [] (terminal)
     ```
   - Throws hard error with descriptive message on invalid transition
   - Requires `--reason` for: failed, skipped, and verifying->investigating (retry)
   - Appends to `transitions` array: `{ from, to, timestamp, reason }`
   - Updates `durations`: sets `exited` and `seconds` on current state, creates `entered` for new state
   - If transitioning to `failed` or `skipped`, sets `failure_reason` field
   - If transitioning `verifying -> investigating` (retry), increments `current_attempt`
   - Updates `state` and `updated` timestamp
   - Writes back to file (preserve body content below frontmatter)
   - Returns JSON: `{ from, to, timestamp, path }`

3. `ticket list <session-dir> [--state <state>]`
   - Lists all `.md` files in `<session-dir>/tickets/`
   - For each: extracts frontmatter, returns `{ number, slug, state, path }`
   - If `--state` filter provided, only return matching tickets
   - Sort by number ascending (FIFO)
   - Returns JSON array

4. `ticket next <session-dir>`
   - Calls ticket list internally with `--state queued`
   - Returns the first queued ticket (lowest number) or `null` if none
   - Returns JSON: `{ path, number, slug }` or `{ path: null }`

5. `session create <base-dir> [--name <name>]`
   - `<base-dir>` is the `.fixme/sessions/` directory
   - If no `--name`: auto-generate as `fix-YYYYMMDD-HHmmss`
   - If name collision (directory exists): append `-2`, `-3`, etc.
   - Creates: `<base-dir>/<name>/`, `<base-dir>/<name>/tickets/`, `<base-dir>/<name>/assets/`
   - Reads the session template from `~/.claude/skills/fixme/templates/session.md`
   - Fills placeholders: `{SESSION_NAME}`, `{TIMESTAMP}`
   - Writes `<base-dir>/<name>/session.md`
   - Returns JSON: `{ path, name, created }`

6. `session list <base-dir>`
   - Lists all session directories in `<base-dir>`
   - For each: reads session.md frontmatter, counts tickets by state
   - Returns JSON array: `[{ name, path, created, ticket_counts: { queued: N, ... } }]`

7. `session summary <session-dir>`
   - Reads all tickets in session
   - Computes: total tickets, counts by terminal state (done, failed, skipped), counts by active state
   - Computes total session duration (from session.md created to now)
   - Per-ticket breakdown: `{ number, slug, state, total_seconds }`
   - Writes summary stats to session.md frontmatter (adds `completed`, `duration_seconds`, `tickets_done`, `tickets_failed`, `tickets_skipped`)
   - Returns JSON with full summary

**Frontmatter parser:** Build a simple line-by-line YAML frontmatter parser that handles:
- `---` delimiters
- Simple key-value pairs (string, number, boolean, null)
- Arrays (both inline `[a, b]` and multiline `- item` syntax)
- Nested objects (using indentation, max 2 levels for `durations` and `transitions`)
- Transition log entries as array of objects: `{ from, to, timestamp, reason }`
- String values with special characters (quotes preserved)

**Frontmatter serializer:** Writes frontmatter back preserving:
- Field ordering from the parsed original (new fields appended)
- Proper YAML formatting for nested structures
- Correct indentation for arrays of objects

**Error handling:** All errors output JSON `{ error: "message" }` to stdout and exit with code 1. Never throw unhandled exceptions.

**Do NOT:**
- Use any npm packages (pure Node.js: fs, path, process)
- Build a full YAML parser -- handle only the known ticket/session schema shapes
- Create an index file or shared mutable state (queue derived from filesystem)

Extend `fixme-tools.cjs` (created in Task 1) with three context subcommands. These implement BROW-01: project context auto-detection.

1. `context detect [--project-dir <dir>]`
   - Defaults to current working directory if `--project-dir` not provided
   - Scans multiple sources in this priority order:
     a. `package.json`: extract `scripts.dev`, `scripts.build`, `scripts.test`, `scripts.lint`, detect framework from dependencies (next.js, nuxt, angular, svelte, vue, react), detect test runner from devDependencies (vitest, jest, mocha)
     b. Config files (`vite.config.ts`, `vite.config.js`, `next.config.js`, `next.config.mjs`, `next.config.ts`, `webpack.config.js`): detect HMR capability (Vite and Next.js have HMR by default)
     c. `.env` / `.env.local`: look for `PORT=` to override default dev server port
   - Build context object:
     ```
     dev_server: { command, url, hmr }
     build: { command }
     test: { runner, command, filter_by_file, filter_by_name, init_command }
     lint: { command }
     framework: string
     detected_from: string[]
     detected_at: ISO timestamp
     ```
   - For test filters, generate patterns based on detected runner:
     - vitest: `filter_by_file: "yarn test {file}"`, `filter_by_name: "yarn test -t '{name}'"`
     - jest: same pattern as vitest (compatible CLI)
     - mocha: `filter_by_file: "yarn test -- {file}"`, `filter_by_name: "yarn test -- --grep '{name}'"`
   - Default dev server URL: `http://localhost:3000` (or use PORT from .env if found)
   - Use `yarn` as the package manager prefix for commands (per user's preference in CLAUDE.md)
   - Returns JSON with the full context object
   - Does NOT write to disk (user must confirm first -- the orchestrator handles confirmation flow)

2. `context save <yaml-content> [--project-dir <dir>]`
   - Writes the confirmed project context to `<project-dir>/.fixme/project-context.yaml`
   - Creates `.fixme/` directory if it doesn't exist
   - Input: YAML string via stdin or as argument
   - Actually, simpler: accept JSON via `--data '{...}'` flag and serialize to YAML
   - Write a simple JSON-to-YAML serializer for the known shallow schema (max 2 levels)
   - Returns JSON: `{ path, saved: true }`

3. `context load [--project-dir <dir>]`
   - Reads `.fixme/project-context.yaml`
   - Parses YAML back to object (reuse the shallow YAML parser)
   - Returns JSON with the context object
   - If file doesn't exist, returns `{ error: "No project context found. Run 'context detect' first." }` with exit code 1

**YAML serializer for context (JSON -> YAML string):**
Simple and purpose-built for the known schema:
- Top-level keys become YAML keys
- Nested objects (dev_server, build, test, lint) become indented key-value pairs
- Arrays (detected_from) become `- item` syntax
- Null values become empty (key with no value)
- String values with special characters get quoted

**Do NOT:**
- Parse CLAUDE.md for context (per user decision: "Agents already inherit CLAUDE.md through normal Claude Code mechanisms -- project context only captures supplementary operational info")
- Include coding conventions, auth credentials, or language detection
- Write to disk without explicit save command (no silent writes, per user decision)
